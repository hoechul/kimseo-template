# UI Project Kickoff Pack

이 문서는 `kim-secretary` 스타일을 새 프로젝트에 이식할 때 사용하는 통합 문서다.  
기획, 디자인, 구현, 검수까지 한 흐름으로 바로 사용할 수 있게 구성했다.

이 문서 하나로 아래 3가지를 대신한다.
- UI/UX 기준서
- 새 프로젝트용 스타터 프롬프트
- 디자인 QA 체크리스트

관련 참고 문서:
- `docs/UI_UX_GUIDELINES.md`
- `docs/UI_STARTER_PROMPT.md`
- `docs/UI_DESIGN_CHECKLIST.md`

## 1. 목표

새 프로젝트를 만들 때 `kim-secretary`의 UI/UX 감각을 유지한다.

단순히 비슷한 색을 쓰는 수준이 아니라 아래를 같이 가져간다.
- 정보 구조
- 화면 밀도
- 카드와 패널의 표면 언어
- 목록/상세/폼의 패턴
- 모바일 재구성 방식
- 상태 처리 방식

## 2. 제품 인상

반드시 아래 인상을 유지한다.

- 업무용 SaaS처럼 신뢰감이 있어야 한다.
- 차갑고 건조한 관리자 화면이 아니라 정돈된 업무 공간처럼 보여야 한다.
- 정보량은 많아도 피로하지 않아야 한다.
- 부드럽고 안정적인 카드 중심 인터페이스여야 한다.
- 과한 브랜딩보다는 실무적인 세련됨이 우선이다.

핵심 키워드:
- 차분함
- 정돈감
- 친절한 밀도
- 신뢰감
- 부드러운 업무 도구

## 3. 비주얼 원칙

### 3.1 색상

- 배경은 순백 대신 따뜻한 아이보리/베이지 계열
- 메인 컬러는 청록 계열
- 포인트 컬러는 약한 골드/샌드 계열
- 성공/경고/위험은 상태 표현에서만 명확히 사용
- 전체 채도는 낮게 유지

### 3.2 폰트

- 기본 폰트는 `Pretendard`
- 제목은 짧고 진하게
- 설명은 짧고 부드럽게
- 메타 정보는 작은 크기와 약한 색으로 표현

### 3.3 형태

- 주요 카드/패널은 큰 라운드 사용
- 기본 라운드 감각은 `1.5rem~2rem`
- 버튼/입력/탭도 둥글게 통일
- 카드와 패널은 soft glass 느낌 유지

### 3.4 표면

- 반투명 배경
- 얕은 border
- 넓고 은은한 그림자
- backdrop blur

요약하면:
- 딱딱한 flat admin UI 금지
- 유리처럼 부드러운 업무 패널 구조 유지

## 4. 페이지 구조

모든 핵심 페이지는 가능하면 이 순서를 따른다.

1. `PageShell`
2. `PageHeader`
3. `StatsGrid` 또는 `PageToolbar`
4. 본문 콘텐츠
5. 생성/수정 UI

### 4.1 헤더 규칙

- 제목
- 짧은 설명
- 우측 주요 액션
- 필요 시 breadcrumb
- 필요 시 상태/보조 배지

### 4.2 본문 규칙

- 긴 한 덩어리 대신 섹션 단위로 분리
- 섹션마다 카드 또는 패널로 감싸기
- 숫자, 상태, 리스트, 상세 정보의 위계를 분명히 나누기

## 5. 화면 유형별 규칙

### 5.1 목록 페이지

- 검색 + 필터 + 상태 탭 + 목록 구조
- 데스크톱은 테이블
- 모바일은 카드 리스트
- 행 전체 클릭 가능
- 상태는 배지로 표시
- 요약 통계는 3~4개 카드로 제한

### 5.2 상세 페이지

- 상단에 제목, 상태, 주요 액션
- 본문은 여러 상세 카드 섹션으로 분리
- 관련 리소스/기록/연관 데이터 분리

### 5.3 폼 페이지

- 단순 입력은 다이얼로그
- 복잡한 입력은 별도 페이지
- 폼은 2열 그리드를 기본으로 사용
- 중요한 필드는 전체 폭 사용
- 저장/취소 액션은 일관된 위치 유지

### 5.4 캘린더/운영 화면

- 날짜 이동, 보기 전환, 생성 액션을 상단 툴바에 통합
- 필터는 사이드 패널 또는 모바일 시트로 제공
- 단축키가 있으면 힌트를 UI에 드러냄

## 6. 컴포넌트 규칙

### 6.1 버튼

- 기본 버튼은 채워진 primary
- 보조 액션은 `outline`
- 문맥성 액션은 `ghost`
- 충분한 클릭 영역 유지

### 6.2 입력

- 부드러운 배경과 테두리
- 명확한 focus ring
- 검색 입력은 좌측 아이콘 허용
- 검색 가능한 선택은 combobox/popover 패턴 우선

### 6.3 카드

- 기본 UI 표면은 카드
- 단순 박스가 아니라 깊이감 있는 패널이어야 함
- 정보 밀도는 카드 내부 spacing으로 제어

### 6.4 테이블

- 데스크톱에서만 표 구조 사용
- 모바일은 카드형으로 재배치
- hover, sort, active 상태가 보여야 함

### 6.5 배지

- 상태는 텍스트만 두지 말고 배지화
- 색은 보조 수단, 텍스트는 필수

## 7. 상태 UX

반드시 아래 3개를 공통 컴포넌트로 준비한다.

- 로딩 상태
- 오류 상태
- 빈 상태

규칙:
- 빈 상태에는 다음 행동 제안 포함
- 오류 상태에는 재시도 버튼 포함
- 상태 UI도 일반 카드 톤과 일관되게 설계

## 8. 반응형 원칙

- 모바일은 축소가 아니라 재구성
- 테이블은 카드로 변환
- 사이드 필터는 모달/시트로 전환
- 상단 액션은 세로 적층 허용
- 플로팅 UI는 모바일 화면을 과도하게 가리지 않게 제한

## 9. 카피 원칙

- 제목은 짧고 명확하게
- 설명은 “여기서 무엇을 하는지”를 바로 알게 작성
- 오류 메시지는 문제 + 다음 행동을 함께 제시
- 빈 상태 문구는 건조하지 않되 실무적 톤 유지
- 핵심 데이터, 금액, 계약, 오류 안내에는 가벼운 농담 금지

## 10. 새 프로젝트용 스타터 프롬프트

아래 프롬프트를 새 프로젝트 시작 시 그대로 사용할 수 있다.

```md
이 프로젝트의 UI/UX는 `kim-secretary` 스타일을 기준으로 설계해줘.

반드시 아래 원칙을 따라:

- 업무용 SaaS처럼 신뢰감 있고 정돈된 화면이어야 한다.
- 차갑고 건조한 관리자 화면이 아니라 부드럽고 안정적인 업무 공간처럼 보여야 한다.
- 기본 폰트는 Pretendard를 사용한다.
- 배경은 따뜻한 밝은 톤, 메인 컬러는 청록 계열, 포인트는 약한 골드 계열을 사용한다.
- 카드와 패널은 반투명 배경, 옅은 보더, 큰 라운드, 은은한 그림자, blur를 사용한다.
- 페이지 구조는 `PageShell > PageHeader > Toolbar/Stats > Content` 순서로 설계한다.
- 목록 페이지는 데스크톱 테이블 / 모바일 카드 구조로 만든다.
- 상세 페이지는 긴 문서형이 아니라 여러 카드 섹션으로 나눈다.
- 단순 CRUD는 다이얼로그, 복잡한 입력은 별도 폼 페이지로 분리한다.
- 로딩/오류/빈 상태 UI를 공통 패턴으로 설계한다.
- 모바일은 축소가 아니라 재배치 방식으로 설계한다.
- hover, focus, active 상태가 모두 명확해야 한다.

금지 사항:
- 순백 배경 + 강한 원색 버튼
- 각진 카드 위주의 딱딱한 관리자 화면
- 모바일에서 축소된 테이블
- 일관성 없는 페이지 헤더
- 텍스트 한 줄로 끝나는 빈 상태/오류 상태

가능하면 공통 레이아웃, 공통 카드, 공통 상태 컴포넌트까지 함께 설계해줘.
```

```md
반응형 우선순위:
```

## 12. 디자인 체크리스트

### 12.1 기본 인상

- 신뢰감 있는 업무 도구처럼 보이는가
- 너무 차갑거나 너무 화려하지 않은가
- 첫 화면에서 정보 계층이 읽히는가

### 12.2 비주얼

- Pretendard 적용 완료
- 따뜻한 배경 톤 적용
- 청록 primary 적용
- radius, border, shadow, blur 규칙 일관성 유지

### 12.3 구조

- 모든 페이지에 공통 헤더 문법이 있는가
- 헤더 > 툴바/통계 > 본문 순서가 유지되는가
- 본문 폭이 과도하게 넓지 않은가

### 12.4 목록

- 검색/필터/상태 탭 배치가 정리되어 있는가
- 데스크톱은 테이블, 모바일은 카드인가
- 행 전체 클릭 또는 명확한 진입 동선이 있는가
- 상태에 배지가 사용되는가

### 12.5 상세

- 상세 정보가 섹션별 카드로 나뉘는가
- 핵심 정보가 상단에 우선 배치되는가
- 관련 데이터와 기록이 뒤섞이지 않는가

### 12.6 폼

- 여백이 충분한가
- 라벨과 입력 관계가 명확한가
- 긴 폼이 적절히 2열화되어 있는가
- 저장/취소 액션 위치가 일관적인가

### 12.7 상태

- 로딩 UI 있음
- 오류 UI 있음
- 빈 상태 UI 있음
- 재시도 및 다음 행동 유도 있음

### 12.8 반응형

- 모바일이 축소판이 아닌가
- 테이블이 카드로 바뀌는가
- 필터가 모바일에서 시트/모달로 바뀌는가
- 플로팅 UI가 모바일 화면을 과도하게 가리지 않는가

### 12.9 인터랙션

- hover 보임
- focus ring 보임
- active/selected 구분 가능
- 클릭 영역 충분

## 13. 금지 사항

- 순백 배경 중심의 건조한 관리자 UI
- 각진 패널과 얇은 테두리만 있는 화면
- 원색 남용
- 모바일에서 읽기 어려운 dense table
- 아이콘만 있고 설명 없는 상태 표현
- 페이지마다 다른 문법의 액션 배치

## 14. 권장 구현 순서

1. 전역 색상/폰트/표면 토큰 설정
2. 공통 레이아웃 컴포넌트 작성
3. 버튼/입력/카드/배지/테이블 스타일 통일
4. 목록/상세/폼/상태 컴포넌트 작성
5. 모바일 재구성
6. 실제 페이지 적용
7. 체크리스트로 QA

## 15. 최종 기준

완성된 화면은 아래에 가까워야 한다.

- 보기 좋은 관리자 화면
- 실제로 일하기 편한 운영 도구
- 차분하지만 심심하지 않은 업무 공간
- 구조가 정리되어 있어서 오래 봐도 덜 피곤한 제품

이 문서를 새 프로젝트의 UI 킥오프 기준 문서로 사용한다.
