# UI Design Checklist

새 프로젝트에서 `kim-secretary` 스타일을 적용할 때 사용하는 체크리스트다.

## 1. 기본 인상

- 화면이 업무용 SaaS처럼 신뢰감 있게 보이는가
- 과하게 차갑거나 딱딱하지 않고, 정돈된 업무 공간처럼 보이는가
- 첫 화면에서 정보 계층이 한눈에 구분되는가
- 템플릿 느낌보다 실제 서비스 느낌이 강한가

## 2. 비주얼 토큰

- 전역 폰트가 `Pretendard`로 설정되어 있는가
- 배경이 순백이 아니라 따뜻한 밝은 톤인가
- primary가 청록 계열로 통일되어 있는가
- 포인트 컬러가 과하지 않고 보조적으로만 쓰이는가
- border, shadow, blur, radius 값이 전역적으로 일관적인가

## 3. 표면과 형태

- 주요 카드와 패널이 충분히 둥근가
- 카드가 단순 흰 박스가 아니라 soft surface처럼 보이는가
- 그림자가 좁고 진하지 않고 넓고 은은한가
- 카드 간 밀도가 과하지 않고 숨 쉴 공간이 있는가

## 4. 페이지 구조

- 각 페이지가 `헤더 > 툴바/통계 > 본문` 순서를 따르는가
- 헤더에 제목, 설명, 주요 액션이 명확히 배치되는가
- 페이지마다 헤더 구조가 제각각이지 않은가
- 본문 폭이 너무 넓게 풀리지 않았는가

## 5. 목록 페이지

- 검색창 위치가 일관적인가
- 필터와 상태 탭이 한 덩어리로 정돈되어 보이는가
- 데스크톱에서 테이블이 읽기 쉬운가
- 모바일에서 테이블이 카드 구조로 재설계되었는가
- 목록 행 전체가 클릭 가능하거나, 진입 동선이 명확한가
- 상태 표현에 배지가 사용되는가

## 6. 상세 페이지

- 상세 정보가 긴 텍스트 한 덩어리가 아니라 섹션으로 분리되는가
- 중요한 정보가 상단에 먼저 보이는가
- 관련 리소스, 기록, 부가 정보가 구획별로 나뉘는가
- 편집/삭제/후속 액션이 찾기 쉬운 위치에 있는가

## 7. 폼 UX

- 폼이 충분한 여백을 가지고 있는가
- 라벨과 입력 간 관계가 명확한가
- 긴 폼은 2열 그리드와 전체 폭 필드를 적절히 섞었는가
- 필수/선택 입력이 분명한가
- 검색형 선택은 일반 select보다 더 나은 방식이 필요한지 검토했는가
- 저장/취소 버튼 위치가 일관적인가

## 8. 상태 처리

- 로딩 상태 UI가 있는가
- 오류 상태 UI가 있는가
- 빈 상태 UI가 있는가
- 빈 상태에서 다음 행동을 유도하는가
- 오류 상태에서 재시도 동선이 있는가

## 9. 인터랙션

- hover 상태가 보이는가
- focus ring이 충분히 명확한가
- active/selected 상태가 분명한가
- 클릭 영역이 충분히 넓은가
- 아이콘만으로 의미를 전달하지 않는가

## 10. 반응형

- 모바일에서 정보 구조가 다시 설계되었는가
- 상단 액션이 모바일에서 무너지지 않는가
- 필터 사이드바가 모바일에서 모달/시트로 전환되는가
- 고정 패널이나 플로팅 UI가 모바일 화면을 과도하게 가리지 않는가
- 작은 화면에서도 텍스트와 숫자가 읽기 쉬운가

## 11. 카피

- 페이지 제목이 짧고 명확한가
- 설명 문장이 지나치게 길지 않은가
- 오류 메시지가 문제와 행동을 함께 안내하는가
- 빈 상태 문구가 건조하지 않으면서도 실무적 톤을 유지하는가
- 핵심 데이터 영역에서는 농담이나 과한 감성 문구를 쓰지 않았는가

## 12. 일관성

- 버튼 스타일이 페이지마다 달라지지 않는가
- 카드 스타일이 페이지마다 달라지지 않는가
- 배지, 입력, 테이블 규칙이 공통화되어 있는가
- spacing scale이 일정한가
- radius scale이 일정한가

## 13. 최종 QA

- 첫 화면 스크린샷만 봐도 브랜드 톤이 일관적인가
- 3개 이상의 핵심 페이지를 나란히 봤을 때 같은 제품처럼 보이는가
- 모바일/데스크톱 모두에서 완성도가 유지되는가
- "보기 좋다"보다 "실제로 일하기 편하다"에 더 가깝게 느껴지는가

## 간단 판정 기준

아래 5개 중 4개 이상 만족하면 스타일 유지가 잘 된 편이다.

- 따뜻한 밝은 배경과 청록 중심 톤이 유지된다
- 카드와 패널이 부드럽고 일관된 표면 언어를 가진다
- 헤더와 툴바 구조가 정리되어 있다
- 목록/폼/상세/상태 화면이 모두 공통 문법을 따른다
- 모바일이 축소판이 아니라 재설계된 화면이다
