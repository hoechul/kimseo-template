# KIM Secretary UI/UX 가이드

이 문서는 `kim-secretary` 프로젝트의 실제 구현을 기준으로 추출한 UI/UX 규칙이다. 앞으로 새 프로젝트를 만들 때 "비슷하게"가 아니라, 이 문서의 구조와 원칙을 기본값으로 삼는다.

기준 소스:
- `src/app/globals.css`
- `src/components/page-shell.tsx`
- `src/components/sidebar.tsx`
- `src/components/ui/*.tsx`
- `src/app/dashboard/**/*`
- `src/components/*-table.tsx`, `*-dialog.tsx`, `*-form.tsx`

## 1. 제품 인상

- 업무용 SaaS답게 신뢰감이 있어야 한다.
- 딱딱한 B2B 화면처럼 보이되, 차갑지 않게 만든다.
- "관리 툴"이 아니라 "정리된 업무 공간"처럼 보여야 한다.
- 강한 장식 대신, 부드러운 유리 질감과 큰 라운드, 은은한 그림자로 밀도를 만든다.
- 정보량은 많지만 피로감은 낮아야 한다.

핵심 키워드:
- 차분함
- 정돈감
- 친절한 밀도
- 업무 중심
- 과하지 않은 브랜딩

## 2. 비주얼 시스템

### 2.1 색상

- 배경은 순백이 아니라 따뜻한 아이보리 톤을 쓴다.
- 브랜드 메인은 청록 계열을 사용한다.
- 보조 포인트는 골드/샌드 계열을 아주 약하게 섞는다.
- 성공/경고/위험 상태는 배지와 통계 카드에서만 분명하게 드러낸다.
- 전체 화면은 저채도 중심으로 유지한다.

구현 기준:
- 배경: `--background: oklch(0.985 0.01 92)`
- 본문: `--foreground: oklch(0.255 0.04 208)`
- 메인 컬러: `--primary: oklch(0.47 0.102 205)`
- 보조 배경: `--secondary`, `--muted`, `--accent`
- 전역 배경 효과: `body`의 radial-gradient + linear-gradient

### 2.2 타이포그래피

- 기본 폰트는 반드시 `Pretendard`를 사용한다.
- 제목은 굵고 짧게, 설명은 짧고 부드럽게 쓴다.
- 데이터 라벨은 작고 차분하게, 값은 더 크고 진하게 보여준다.
- 과도한 대문자 스타일은 피하고, 필요한 경우에만 tracking을 준다.

구현 기준:
- 전역 폰트: `src/app/globals.css`
- 큰 제목: `text-xl`, `md:text-2xl`, `font-semibold`, `tracking-tight`
- 보조 설명: `text-sm`, `text-muted-foreground`
- 메타 라벨: `text-xs`, `uppercase`, `tracking-[0.16em]` 또는 `tracking-[0.22em]`

### 2.3 형태

- 기본 코너는 둥글어야 한다. 작은 입력도 각지지 않는다.
- 주요 컨테이너는 `1.5rem` 이상 라운드를 유지한다.
- 화면의 대부분은 직사각형보다 "소프트 카드"처럼 보여야 한다.

구현 기준:
- 카드 기본: `rounded-[1.5rem]`
- 큰 패널: `rounded-[1.75rem]`, `rounded-[2rem]`
- 버튼: `rounded-xl`
- 칩/배지: `rounded-full`

### 2.4 그림자와 레이어

- 진한 그림자 대신 넓고 흐린 그림자를 사용한다.
- 카드와 패널은 `bg-card/75~90` + `backdrop-blur` 조합을 기본으로 한다.
- 정보 우선순위는 색보다 레이어와 간격으로 표현한다.

재사용 클래스:
- `.surface-panel`
- `.surface-subtle`
- `.page-title-gradient`

## 3. 레이아웃 원칙

### 3.1 전체 구조

- 데스크톱은 `좌측 사이드바 + 중앙 메인 + 우측 플로팅 보조패널` 구조를 기본으로 한다.
- 모바일은 사이드바를 상단 헤더 + 시트 메뉴로 바꾼다.
- 메인 콘텐츠 폭은 너무 넓게 풀지 말고 `max-w-7xl` 안에서 관리한다.

구현 기준:
- `src/app/dashboard/layout.tsx`
- `Sidebar`
- `MobileSidebar`
- `ChatPanel`

### 3.2 여백

- 페이지 단위 여백은 넉넉해야 한다.
- 섹션 간 간격은 좁게 붙이지 않는다.
- 카드 내부 패딩은 작아도 숨 막히지 않게 유지한다.

구현 기준:
- 페이지 외곽: `px-4 py-4 md:px-6 md:py-6`
- 페이지 스택: `space-y-6` 또는 `space-y-5`
- 카드 패딩: `p-4`, `p-5`, `p-6`

## 4. 네비게이션 패턴

### 4.1 사이드바

- 사이드바는 단순 링크 목록이 아니라 "현재 일하는 공간"처럼 보여야 한다.
- 브랜드, 메뉴 섹션, 사용자 프로필의 3단 구조를 유지한다.
- 활성 메뉴는 채워진 메인 컬러 배경으로 명확히 구분한다.
- 비활성 메뉴는 텍스트보다 hover 반응으로 존재감을 준다.

구현 기준:
- 사이드바 폭: `w-72`, 축소 시 `w-20`
- 컨테이너: 반투명 배경 + blur + 라운드 2rem
- 활성 메뉴: `bg-primary/95 text-primary-foreground`

### 4.2 모바일

- 모바일은 상단 고정 헤더에서 현재 페이지 이름을 먼저 보여준다.
- 메뉴는 좌측 시트로 연다.
- 데스크톱 구조를 억지로 축소하지 말고, 요약 정보 위주로 재배치한다.

## 5. 페이지 구조 규칙

모든 주요 페이지는 이 순서를 기본으로 한다.

1. `PageShell`
2. `PageHeader`
3. `StatsGrid` 또는 `PageToolbar`
4. 본문 리스트/캘린더/상세 섹션
5. 생성/수정용 다이얼로그 또는 별도 폼 페이지

### 5.1 헤더

- 제목, 짧은 설명, 우측 액션 버튼 조합을 기본으로 한다.
- 설명 문장은 길게 쓰지 않는다.
- 필요하면 breadcrumb, eyebrow, accessory를 추가한다.
- 페이지 톤을 부드럽게 만들기 위해 유머 문구를 한 줄 섞는 방식은 유지 가능하다.

구현 기준:
- `PageHeader`
- `PageFunMessage`

### 5.2 통계 카드

- 첫 화면 인지는 숫자가 아니라 구조다. 통계 카드는 3~4개까지만 둔다.
- 각 카드에는 라벨, 핵심 값, 짧은 설명, 아이콘을 둔다.
- 색상은 상태 구분이 필요한 카드에만 쓴다.

구현 기준:
- `StatsGrid`
- `StatCard`
- tone: `default`, `info`, `success`, `warning`, `danger`

## 6. 화면 유형별 규칙

### 6.1 목록 페이지

- 기본 패턴은 `검색 + 상태 필터/탭 + 보조 필터 + 목록`이다.
- 데스크톱은 테이블, 모바일은 카드 리스트로 분기한다.
- 목록 행 전체를 클릭 가능하게 만든다.
- 행 안에 작은 액션 버튼을 남발하지 않는다.

구현 기준:
- `project-table.tsx`
- `customer-table.tsx`
- `employee-table.tsx`
- `quotations/page.tsx`
- `contracts/page.tsx`

### 6.2 상세 페이지

- 상단에는 breadcrumb, 제목, 상태, 주요 액션을 배치한다.
- 상세 정보는 한 덩어리 긴 문서가 아니라 여러 카드/섹션으로 나눈다.
- 핵심 정보, 관련 리소스, 활동/기록을 분리해 읽히게 한다.

### 6.3 생성/수정 화면

- 단순한 엔터티는 다이얼로그 안에서 처리한다.
- 정보량이 많거나 연산/미리보기가 있으면 별도 페이지 폼으로 분리한다.
- 폼은 2열 그리드를 기본으로 하되, 중요한 필드는 전체 폭을 사용한다.

구현 기준:
- 다이얼로그형: `project-dialog.tsx`
- 폼 페이지형: `task-form.tsx`, `quotation-form.tsx`

### 6.4 캘린더형 화면

- 헤더에서 날짜 이동, 오늘 이동, 뷰 전환, 생성 액션을 한 번에 제공한다.
- 필터 사이드 패널을 통해 표시 대상을 좁힌다.
- 월/주/일 뷰는 공통 조작 규칙을 공유해야 한다.
- 키보드 단축키가 있으면 화면 힌트로 드러낸다.

구현 기준:
- `calendar-header.tsx`
- `calendar-month-view.tsx`
- `calendar-week-view.tsx`
- `calendar-day-view.tsx`
- `employee-filter.tsx`

## 7. 컴포넌트 규칙

### 7.1 버튼

- 기본 버튼은 브랜드 컬러 채움형이다.
- 보조 액션은 `outline`, 문맥성 액션은 `ghost`를 쓴다.
- 버튼은 지나치게 날카롭거나 얇게 만들지 않는다.
- 아이콘은 장식이 아니라 의미 보조용으로만 붙인다.

구현 기준:
- `src/components/ui/button.tsx`

### 7.2 입력

- 입력창은 살짝 채워진 배경과 부드러운 테두리를 쓴다.
- 포커스는 진한 외곽선 대신 ring으로 표현한다.
- 검색창은 좌측 아이콘을 넣되, 밀도를 해치지 않게 한다.
- 검색 가능한 선택은 `Popover + Command` 조합을 우선 사용한다.

구현 기준:
- `src/components/ui/input.tsx`
- `task-form.tsx`
- `project-dialog.tsx`

### 7.3 카드

- 카드 자체가 이 프로젝트의 기본 표면이다.
- 단순 흰 박스 대신 반투명 카드 + 그림자 + 라운드를 유지한다.
- 카드 내 정보는 제목, 설명, 본문 블록의 리듬이 있어야 한다.

구현 기준:
- `src/components/ui/card.tsx`
- `SectionCard`
- `.surface-panel`, `.surface-subtle`

### 7.4 테이블

- 데스크톱 테이블은 카드형 컨테이너 안에 배치한다.
- 헤더는 조용하게, hover 행은 약하게 강조한다.
- 정렬 가능한 컬럼은 헤더 인터랙션으로 처리한다.
- 모바일에서는 같은 정보를 카드 레이아웃으로 다시 설계한다.

구현 기준:
- `src/components/ui/table.tsx`
- `sortable-table-head.tsx`

### 7.5 배지

- 상태 표현은 텍스트만 두지 말고 배지로 묶는다.
- 배지는 작고 둥글게, 너무 진하지 않게 유지한다.
- 상태 의미는 색으로 보조하되, 텍스트를 반드시 남긴다.

구현 기준:
- `src/components/ui/badge.tsx`

## 8. 상호작용 규칙

### 8.1 피드백

- 로딩, 오류, 빈 상태는 반드시 별도 상태 UI를 제공한다.
- 빈 상태는 "없음"만 말하지 말고 다음 행동을 제안한다.
- 오류 상태에는 재시도 버튼을 기본으로 둔다.

구현 기준:
- `LoadingState`
- `ErrorState`
- `EmptyState`

### 8.2 검색과 필터

- 검색은 항상 툴바 상단 좌측에 둔다.
- 검색어 초기화는 별도 버튼으로 제공한다.
- 필터는 토글 버튼/탭/사이드패널 중 하나로 명확히 분리한다.
- 상태 탭에는 개수를 같이 표시한다.

### 8.3 클릭 영역

- 카드형 목록은 카드 전체가 클릭되어야 한다.
- 행 클릭과 내부 버튼 클릭이 충돌하지 않도록 설계한다.
- 작은 텍스트 링크보다 충분한 padding이 있는 버튼을 우선한다.

### 8.4 단축키와 생산성

- 자주 쓰는 화면에는 단축키를 넣을 수 있다.
- 단축키가 있으면 숨기지 말고 UI에 힌트로 노출한다.

예시:
- 캘린더 `M`, `W`, `D`
- 채팅 패널 `Ctrl+/`
- 채팅 패널 닫기 `Esc`

## 9. 카피라이팅 규칙

- 제목은 명사형으로 짧게 쓴다.
- 설명은 사용자가 "여기서 뭘 하는지"를 바로 이해하게 쓴다.
- 오류 문구는 문제와 다음 행동을 같이 준다.
- 빈 상태 문구는 건조하지 않게, 그러나 가볍게 쓴다.
- 업무 툴이라도 전체 톤은 무표정보다 약간 인간적으로 유지한다.

주의:
- 유머 문구는 헤더 보조 설명 수준에서만 사용한다.
- 핵심 데이터, 금액, 계약, 오류 안내에는 유머를 섞지 않는다.

## 10. 반응형 규칙

- 모바일은 데스크톱 UI를 축소하지 말고 재구성한다.
- 테이블은 모바일에서 카드 리스트로 바꾼다.
- 필터 사이드바는 모바일에서 모달/시트로 옮긴다.
- 상단 액션 버튼은 모바일에서 세로 적층을 허용한다.
- floating 보조 UI는 모바일 viewport를 가리지 않게 높이와 폭을 제한한다.

## 11. 구현 체크리스트

새 프로젝트에서 이 스타일을 재현하려면 아래를 기본값으로 적용한다.

- `Pretendard`를 전역 폰트로 설정한다.
- 따뜻한 밝은 배경 + 청록 메인 컬러 + 약한 골드 포인트를 쓴다.
- `surface-panel`, `surface-subtle` 같은 반투명 카드 표면을 만든다.
- 페이지는 `PageShell > PageHeader > Toolbar/Stats > Content` 순서를 따른다.
- 목록 페이지는 데스크톱 테이블 + 모바일 카드 이중 구조를 가진다.
- 생성/수정은 단순하면 다이얼로그, 복잡하면 별도 페이지 폼으로 분리한다.
- 로딩/오류/빈 상태를 공통 컴포넌트로 통일한다.
- 필터와 검색은 헤더 아래 툴바에 모은다.
- 큰 라운드와 넉넉한 spacing을 유지한다.
- hover, focus, active 상태가 반드시 보이게 만든다.

## 12. 새 프로젝트에 그대로 가져갈 기준

권장 우선순위:

1. `globals.css`의 색상 토큰과 배경 처리
2. `page-shell.tsx`의 페이지 구조 컴포넌트
3. `button`, `input`, `card`, `badge`, `table` 스타일
4. `sidebar.tsx`의 내비게이션 구조
5. 목록/상세/폼/캘린더 패턴

복제 기준:
- "같은 느낌"이 아니라 "같은 정보 구조와 표면 언어"를 복제한다.
- 보이는 요소보다 간격, 계층, 상태 표현, 반응형 재구성을 더 우선해서 맞춘다.

## 13. 금지 사항

- 순백 배경 + 진한 원색 버튼 조합
- 각진 카드와 얇은 테두리만으로 끝내는 화면
- 모바일에서 테이블을 그대로 축소한 UI
- 페이지마다 전혀 다른 헤더 구조
- 작은 버튼과 좁은 클릭 영역
- 과한 색상 사용
- 설명 없이 아이콘만 남기는 상태 표시
