# UI Starter Prompt

아래 프롬프트는 새 프로젝트를 시작할 때 그대로 붙여 넣어 사용할 수 있는 기본 프롬프트다.  
필요하면 프로젝트 목적과 도메인만 바꿔서 사용한다.

## 범용 버전

```md
이 프로젝트의 UI/UX는 `kim-secretary` 스타일을 기준으로 설계해줘.

반드시 아래 원칙을 따라:

1. 전체 인상
- 업무용 SaaS처럼 신뢰감 있고 정돈된 화면이어야 한다.
- 차갑고 건조한 B2B 화면이 아니라, 부드럽고 안정적인 업무 공간처럼 보여야 한다.
- 정보량은 많아도 피로하지 않아야 한다.

2. 비주얼 스타일
- 기본 폰트는 Pretendard를 사용한다.
- 배경은 순백이 아니라 따뜻한 아이보리/베이지 계열로 구성한다.
- 메인 컬러는 청록 계열을 사용한다.
- 포인트는 골드/샌드 계열을 아주 약하게 섞는다.
- 카드와 패널은 반투명 배경, 옅은 보더, 큰 라운드, 은은한 그림자, backdrop blur를 사용한다.
- 전체적으로 채도를 낮게 유지하고, 상태 표현에서만 색을 분명히 사용한다.

3. 형태와 간격
- 주요 카드/패널은 1.5rem~2rem 라운드를 사용한다.
- 버튼, 입력창, 탭, 필터도 각지지 않게 둥글게 만든다.
- 간격은 촘촘하지 않게, 여유 있고 정돈되게 잡는다.
- 정보 우선순위는 진한 색보다 간격과 레이어로 표현한다.

4. 페이지 구조
- 기본 페이지 구조는 `PageShell > PageHeader > Toolbar/Stats > Content` 순서로 설계한다.
- 헤더에는 제목, 짧은 설명, 우측 액션 버튼을 둔다.
- 목록 페이지는 검색/필터/상태 탭과 목록 본문으로 나눈다.
- 상세 페이지는 긴 한 덩어리 대신 여러 카드 섹션으로 나눈다.
- 단순 생성/수정은 다이얼로그, 복잡한 입력은 별도 폼 페이지로 분리한다.

5. 목록/데이터 UI
- 데스크톱은 테이블, 모바일은 카드 리스트로 재구성한다.
- 목록 행 전체가 클릭 가능해야 한다.
- 상태는 텍스트만 두지 말고 배지로 보여준다.
- 요약 통계는 3~4개의 카드로 보여준다.

6. 입력 UX
- 입력창은 부드러운 배경과 테두리, 명확한 focus ring을 가진다.
- 검색 가능한 선택은 가능하면 combobox/popover + command 패턴으로 만든다.
- 긴 폼은 2열 그리드를 기본으로 하고, 중요한 필드는 전체 폭을 사용한다.

7. 상태 UX
- 로딩, 오류, 빈 상태를 반드시 별도 UI로 설계한다.
- 빈 상태에는 다음 행동을 유도하는 문구나 버튼을 넣는다.
- 오류 상태에는 재시도 액션을 기본 제공한다.

8. 반응형
- 모바일은 데스크톱 화면을 축소하지 말고 재배치한다.
- 테이블은 모바일에서 카드 구조로 바꾼다.
- 사이드 필터는 모바일에서 모달/시트로 전환한다.

9. 인터랙션
- hover, focus, active 상태가 모두 보여야 한다.
- 작은 텍스트 링크보다 충분한 클릭 영역을 가진 버튼을 우선한다.
- 필요하면 키보드 단축키 힌트를 UI에 자연스럽게 노출한다.

10. 금지 사항
- 순백 배경 + 강한 원색 버튼 조합
- 각진 카드와 딱딱한 관리자 화면
- 모바일에서 축소된 테이블
- 페이지마다 일관성 없는 헤더 구조
- 빈 상태/오류 상태를 텍스트 한 줄로 끝내는 방식

구현할 때는 shadcn/ui + Tailwind 스타일의 컴포넌트 체계를 기준으로 맞춰줘.
가능하면 공통 레이아웃, 공통 카드, 공통 툴바, 공통 상태 컴포넌트까지 함께 설계해줘.
결과물은 실제 서비스 화면처럼 보여야 하고, 템플릿 느낌이 나면 안 된다.
```

## 코딩 에이전트용 버전

```md
이 프로젝트의 UI는 `kim-secretary` 스타일을 기준으로 구현해.

핵심 규칙:
- Pretendard 폰트 사용
- 따뜻한 밝은 배경 + 청록 primary + 약한 골드 포인트
- 큰 라운드(1.5rem 이상), soft glass 카드, 옅은 border, 넓은 shadow
- 페이지 구조는 `PageShell > PageHeader > Stats/Toolbar > Content`
- 목록은 데스크톱 테이블 / 모바일 카드
- 단순 CRUD는 다이얼로그, 복잡한 입력은 별도 폼 페이지
- 상태별 UI(loading/error/empty) 공통화
- hover/focus/active 상태 명확화
- 반응형은 축소가 아니라 재배치

구현 우선순위:
1. 전역 CSS 토큰 정의
2. 공통 레이아웃 컴포넌트 작성
3. 버튼/입력/카드/배지/테이블 스타일 통일
4. 목록/상세/폼 화면 패턴 적용
5. 모바일 최적화

절대 피할 것:
- 기본 system 느낌 그대로의 밋밋한 화면
- 각진 패널
- 과도한 원색 사용
- 모바일에서 읽기 어려운 dense table
- 설명 없는 아이콘 위주 UI
```
